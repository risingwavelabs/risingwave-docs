You can connect RisingWave to an AWS Glue Data Catalog to manage metadata for your Iceberg tables. This allows you to use Glue as a central metastore when creating an Iceberg `SOURCE`, `SINK`, or `CONNECTION` (for internal Iceberg tables).

<Note>
For a complete list of all catalog-related parameters, see the main [Catalog configuration](/iceberg/catalogs#catalog-parameters) page.
</Note>

## Example

To create a sink that uses an AWS Glue catalog, set `catalog.type` to `'glue'` and provide your S3 warehouse path and AWS credentials.

```sql
CREATE SINK my_glue_sink FROM my_mv WITH (
    connector = 'iceberg',
    type = 'upsert',
    primary_key = 'id',
    catalog.type = 'glue',
    warehouse.path = 's3://my-bucket/warehouse/',
    s3.region = 'us-west-2',
    s3.access.key = '...',
    s3.secret.key = '...',
    database.name = 'my_db',
    table.name = 'my_table'
);
```

If you prefer not to specify AWS credentials, you can set `enable_config_load = true` to load them from environment variables instead (self-hosted environments only).

```sql
CREATE SINK my_glue_sink FROM my_mv WITH (
    connector = 'iceberg',
    type = 'upsert',
    primary_key = 'id',
    catalog.type = 'glue',
    warehouse.path = 's3://my-bucket/warehouse/',
    enable_config_load = true, -- Load AWS credentials and region from environment variables
    database.name = 'my_db',
    table.name = 'my_table'
);
```

## Glue-specific parameters

Glue catalogs support additional parameters for separate authentication from S3 credentials.

| Parameter | Description |
|---|---|
| `glue.region` | **Optional**. The AWS region for the Glue catalog. If not specified, defaults to `s3.region`. |
| `glue.access.key` | **Optional**. The AWS access key ID for Glue authentication. If not specified, defaults to `s3.access.key`. |
| `glue.secret.key` | **Optional**. The AWS secret access key for Glue authentication. If not specified, defaults to `s3.secret.key`. |
| `glue.iam_role_arn` | **Optional**. The IAM role ARN to assume for Glue catalog access via STS. When set, the connector assumes the specified role for Glue catalog operations. |

## Separate S3 and Glue credentials

You can use separate credentials for S3 object storage access and Glue catalog operations. This enables tighter security by allowing distinct IAM roles or credentials for each service.

The following example uses different IAM roles for S3 and Glue:

```sql Example
CREATE SINK my_glue_sink FROM my_mv WITH (
    connector = 'iceberg',
    type = 'append-only',
    force_append_only = 'true',
    catalog.type = 'glue',
    catalog.name = 'my_catalog',
    warehouse.path = 's3://my-bucket/warehouse/',
    database.name = 'my_db',
    table.name = 'my_table',
    s3.endpoint = 'https://s3.us-west-2.amazonaws.com',
    s3.region = 'us-west-2',
    s3.access.key = '...',
    s3.secret.key = '...',
    s3.iam_role_arn = 'arn:aws:iam::123456789012:role/S3AccessRole',
    glue.region = 'us-west-2',
    glue.access.key = '...',
    glue.secret.key = '...',
    glue.iam_role_arn = 'arn:aws:iam::123456789012:role/GlueAccessRole'
);
```

In this example:
- S3 operations use the `S3AccessRole` IAM role.
- Glue catalog operations use the `GlueAccessRole` IAM role.

If Glue-specific parameters are not provided, the connector falls back to using the S3 credentials for backward compatibility.
