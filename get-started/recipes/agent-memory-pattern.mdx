**What this does:** Continuously computes state from incoming event streams, queryable at low latency via standard SQL. The pipeline maintains results incrementally — no recomputation at query time.

**When to use this:** Your application needs to read state that reflects real-world events as they happen — user activity, transaction history, system signals — and serve it at low latency.

## Setup

### 1. Start RisingWave

```bash
curl -L https://risingwave.com/sh | sh && risingwave
```

### 2. Ingest events from a source

Events flow in from Kafka, a CDC-connected database, or a webhook endpoint. Here, user activity events come from Kafka:

```sql
CREATE SOURCE user_events (
  user_id    VARCHAR,
  event_type VARCHAR,
  payload    JSONB,
  event_time TIMESTAMPTZ
) WITH (
  connector = 'kafka',
  topic = 'user-activity',
  properties.bootstrap.server = 'localhost:9092',
  scan.startup.mode = 'latest'
) FORMAT PLAIN ENCODE JSON;
```

### 3. Define queryable state as a materialized view

RisingWave maintains this view incrementally as events arrive. The result reflects the latest state — no recomputation triggered at query time.

```sql
CREATE MATERIALIZED VIEW user_context AS
SELECT
  user_id,
  COUNT(*)                                            AS event_count,
  MAX(event_time)                                     AS last_seen,
  ARRAY_AGG(event_type ORDER BY event_time DESC)[1:5] AS recent_events
FROM user_events
WHERE event_time > NOW() - INTERVAL '1 hour'
GROUP BY user_id;
```

### 4. Query state from your application

```python
import psycopg2

conn = psycopg2.connect(host="127.0.0.1", port=4566, user="root", dbname="dev")
conn.autocommit = True
cur = conn.cursor()

def get_user_context(user_id: str) -> dict | None:
    cur.execute(
        "SELECT event_count, last_seen, recent_events FROM user_context WHERE user_id = %s",
        (user_id,)
    )
    row = cur.fetchone()
    if row is None:
        return None
    return {"event_count": row[0], "last_seen": row[1], "recent_events": row[2]}

# Always current — maintained continuously by the event streaming pipeline
context = get_user_context("user-42")
print(context)
```

### 5. Write events back to the pipeline

Applications can emit events that become inputs to the same pipeline — for example, recording a decision result that downstream views consume:

```sql
CREATE TABLE app_actions (
  user_id     VARCHAR,
  action_type VARCHAR,
  result      JSONB,
  recorded_at TIMESTAMPTZ DEFAULT NOW()
);
```

```python
def record_action(user_id: str, action_type: str, result: dict):
```

## Key points

- The materialized view is maintained incrementally by the event streaming pipeline — queries never trigger recomputation
- Connect via the standard PostgreSQL wire protocol; any `psycopg2`, JDBC, or pg-compatible client works
- For push-based patterns (get notified when state changes), see the [Subscription push recipe](/get-started/recipes/subscription-push)
