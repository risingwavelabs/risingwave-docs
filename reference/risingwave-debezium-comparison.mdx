## Introduction

_Debezium is a CDC connector that captures changes from databases and publishes them to Kafka. RisingWave is a streaming database that includes built-in CDC as one of many ingestion paths, and processes captured changes directly inside the system._

They are not direct competitors. Debezium is a component; RisingWave is a system. Whether you need one, the other, or both depends on what you are trying to build.

### Debezium

Debezium is an open-source platform for change data capture. It runs as a set of Kafka Connect source connectors, reading the transaction logs of source databases (PostgreSQL WAL, MySQL binlog, MongoDB oplog, etc.) and publishing each change as an event to a Kafka topic. Debezium is the standard way to stream database changes into a Kafka-centric data platform. It does one thing well: reliably capture every insert, update, and delete and deliver it to Kafka with low latency.

Debezium does not transform data. After changes land in Kafka, a separate system — Flink, ksqlDB, Spark Structured Streaming, or similar — is needed to process them.

### RisingWave

RisingWave is a streaming database with native CDC connectors. It connects directly to source databases via their transaction logs (the same mechanism Debezium uses) and materializes the captured changes into internal tables. From there, SQL-defined materialized views continuously transform and aggregate the data. Results can be queried at low latency over the PostgreSQL wire protocol, or forwarded to downstream systems via sinks.

RisingWave does not require Kafka in the CDC path. The entire pipeline — capture, transform, serve — runs inside a single system.

## Debezium's role in a Kafka-first architecture

Debezium is designed as a building block inside a Kafka-centric platform. It publishes changes to Kafka topics so that many independent downstream consumers — analytics pipelines, search indexers, audit logs, event-driven microservices — can read the same change stream independently at their own pace. If your architecture is already organized around Kafka as a shared event bus for many teams, Debezium fits naturally as the capture layer that feeds it.

Downstream systems that were built to consume the Debezium JSON envelope (`before`/`after`/`op`/`source`) expect that specific format. In such architectures, Debezium and RisingWave are complementary: Debezium writes to Kafka, and RisingWave reads from those Kafka topics as a source.

## What RisingWave does that Debezium does not

**Transformation.** RisingWave processes CDC changes directly with SQL: joins, aggregations, enrichment, deduplication, time-windowed computations. No separate processing engine is required.

**Serving.** Query results are maintained as materialized views and served at 10–20 ms p99 latency via any PostgreSQL client. Debezium has no query interface.

**End-to-end pipeline in one system.** The Debezium stack is: Debezium → Kafka → processing engine → serving database. RisingWave replaces all four with a single deployment.

## How to choose

**Use RisingWave's built-in CDC when:**

- You want to process database changes and serve results without the operational overhead of Kafka and a separate processing engine.
- The CDC pipeline feeds into continuous SQL transformations — enrichment, aggregation, serving — and you want all of it in one system.
- You are replacing a Debezium + Kafka + Flink + serving DB stack with something simpler.

**Use RisingWave + Debezium together when:**

- Your architecture is already organized around Kafka as a shared event bus for many independent consumers, and you want RisingWave to process the same stream for low-latency serving.
- In this case, Debezium writes change events to Kafka, and RisingWave reads from those Kafka topics as a source — getting the best of both: broad fan-out and real-time SQL processing.
