Change data capture (CDC) lets you replicate real-time data changes from operational databases into RisingWave for stream processing and analytics. RisingWave provides **native CDC connectors** for PostgreSQL, MySQL, SQL Server, and MongoDB without requiring Kafka, Kafka Connect, or a separately deployed Debezium service.

## How RisingWave implements CDC

RisingWave reads database change streams differently depending on the source database:

- **PostgreSQL, MySQL, and SQL Server**: RisingWave uses embedded CDC connectors to read upstream change logs directly from the database. Internally, these connectors support Debezium-compatible configuration and Debezium-formatted CDC events, but you do not need to deploy Debezium, Kafka Connect, or Kafka separately.
- **MongoDB**: RisingWave reads MongoDB change streams via the native MongoDB change stream API.

In all cases, changes flow directly from the source database into RisingWave's streaming engine with fault-tolerant recovery. Transactional consistency for multi-table CDC is available when supported by the connector configuration.

## CDC sources and tables

CDC ingestion in RisingWave always materializes into a **table**, because CDC streams contain inserts, updates, and deletes and RisingWave needs table state plus a primary key to apply them correctly.

There are two patterns for creating CDC tables:

**PostgreSQL, MySQL, and SQL Server: create a shared source, then create tables from that source**

Each connector uses a different connector name and slightly different parameters. Select the tab for your database:

<Tabs>
  <Tab title="PostgreSQL">
    ```sql
    CREATE SOURCE pg_src WITH (
      connector = 'postgres-cdc',
      hostname = '127.0.0.1',
      port = '5432',
      username = 'admin',
      password = 'secret',
      database.name = 'mydb',
      schema.name = 'public'  -- Optional, defaults to 'public'
    );

    CREATE TABLE users (
      id INT PRIMARY KEY,
      name VARCHAR,
      email VARCHAR
    ) FROM pg_src TABLE 'public.users';
    ```

    `slot.name` is optional. If omitted, RisingWave generates and manages a unique replication slot automatically.
  </Tab>
  <Tab title="MySQL">
    ```sql
    CREATE SOURCE mysql_src WITH (
      connector = 'mysql-cdc',
      hostname = '127.0.0.1',
      port = '3306',
      username = 'admin',
      password = 'secret',
      database.name = 'mydb',
      server.id = 5001  -- Required; must be unique across all MySQL clients
    );

    CREATE TABLE users (
      id INT PRIMARY KEY,
      name VARCHAR,
      email VARCHAR
    ) FROM mysql_src TABLE 'mydb.users';
    ```
  </Tab>
  <Tab title="SQL Server">
    ```sql
    CREATE SOURCE mssql_src WITH (
      connector = 'sqlserver-cdc',
      hostname = '127.0.0.1',
      port = '1433',
      username = 'sa',
      password = 'secret',
      database.name = 'mydb'
    );

    CREATE TABLE users (
      id INT PRIMARY KEY,
      name VARCHAR,
      email VARCHAR
    ) FROM mssql_src TABLE 'dbo.users';
    ```

    `dbo` is the default SQL Server schema. Replace it with your actual schema name if different.
  </Tab>
</Tabs>

**MongoDB: create the CDC table directly with the connector**

```sql
CREATE TABLE users (
  _id JSONB PRIMARY KEY,
  payload JSONB
) WITH (
  connector = 'mongodb-cdc',
  mongodb.url = 'mongodb://localhost:27017/?replicaSet=rs0',
  collection.name = 'app.users'
);
```

For PostgreSQL, MySQL, and SQL Server, connector-specific options are documented on the individual source pages. Use those pages as the reference for required parameters and exact `CREATE SOURCE` or `CREATE TABLE ... FROM` syntax.

## Shared sources and transactional consistency

When replicating multiple tables from the same PostgreSQL, MySQL, or SQL Server database, use a **shared source**. A shared source establishes one upstream CDC connection and fans out change events to all derived tables.

For PostgreSQL and MySQL, shared sources also preserve multi-table transaction ordering so changes from the same upstream transaction are applied consistently across derived tables. SQL Server also uses the shared source pattern for multi-table ingestion, but it does not support transactional CDC.

Shared sources are supported for PostgreSQL CDC, MySQL CDC, and SQL Server CDC.

## Kafka-based CDC ingestion

You can also ingest CDC events through an event streaming system such as Apache Kafka. In this approach, a separate CDC tool (such as standalone Debezium with Kafka Connect) captures changes from the source database and publishes them as Kafka topics. RisingWave then consumes those topics as a Kafka source, with the message format set to match the CDC tool's output (for example, `FORMAT DEBEZIUM ENCODE JSON`).

This architecture is useful when Kafka is already your central event bus and other downstream systems also need to consume the same change stream. For architectures where RisingWave is the primary consumer, the native CDC connectors are simpler to operate.
