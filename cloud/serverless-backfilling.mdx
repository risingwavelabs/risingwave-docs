## Overview

RisingWave Cloud offers two distinct serverless features for background compute work:

| Feature | Serverless compaction | Serverless backfilling |
| --- | --- | --- |
| **What it does** | Compacts storage files in object storage to keep read performance healthy | Computes the initial state of a materialized view from existing upstream data |
| **Trigger** | Automatically triggered by the storage layer on a continuous basis | Triggered when you run `CREATE MATERIALIZED VIEW` with the serverless option |
| **Where configured** | Checkbox during project/cluster resource setup | **Resources → Backfillings** in the project sidebar |
| **Resource lifecycle** | Managed by RisingWave Cloud; runs continuously as needed | Starts at **0 nodes**; scales up only when eligible pending backfill jobs exist; scales back to 0 when idle |
| **Billing** | Billed per RWU-hour consumed by compactor nodes | Billed per RWU-hour consumed by backfiller nodes, capped by the configured SKU × Replicas setting |

## Configure serverless backfilling

Serverless backfilling settings are separate from the main cluster setup. To access them:

1. Open your project in the RisingWave Cloud console.
2. Select **Resources** in the left sidebar.
3. Click the **Backfillings** tab.
4. Toggle **Serverless backfilling** on, then configure the settings below.

### Backfiller SKU

The **Backfiller SKU** defines the compute size (vCPUs and memory) of each backfiller node. Choose a larger SKU for faster backfill throughput on large upstream tables, or a smaller SKU to minimize cost.

### Replicas

**Replicas** sets the maximum number of backfiller nodes that can run simultaneously. This is the primary cost-control knob:

- The cluster never exceeds **SKU × Replicas** in billable backfiller capacity at any point.
- The actual number of active nodes scales between 0 and the configured replica count based on how many eligible backfill jobs are pending.
- There is no unbounded autoscaling; cost is effectively capped by this setting.

## Resource lifecycle

Serverless backfiller nodes are fully managed and event-driven:

1. **Idle state** – When no eligible backfill jobs are pending, the backfiller scales to **0 nodes** and incurs no compute cost.
2. **Scale-up** – When a `CREATE MATERIALIZED VIEW` statement with the serverless backfill option is executed, the backfiller scales up (up to the configured replica count) to process the job.
3. **Scale-down** – Once all pending jobs complete, the backfiller returns to 0 nodes automatically.

## Activate serverless backfilling for a materialized view

Serverless backfilling applies only to materialized views that explicitly opt in. Use the `WITH` clause when creating the MV:

```sql Example
SET BACKGROUND_DDL = true;

CREATE MATERIALIZED VIEW my_mv
WITH (cloud.serverless_backfill_enabled = true)
AS
SELECT * FROM my_source_table;
```

<Note>
- `cloud.serverless_backfill_enabled = true` must be specified at `CREATE MATERIALIZED VIEW` time. You cannot retroactively enable it for an existing MV.
- Setting `BACKGROUND_DDL = true` is recommended so the DDL statement returns immediately and the backfill runs in the background.
- If serverless backfilling is not enabled in the **Resources → Backfillings** settings, the `cloud.serverless_backfill_enabled` option is ignored and backfill automatically uses the regular streaming nodes instead.
</Note>

## Frequently asked questions

**Does enabling serverless backfilling affect my running cluster?**

No. Backfiller nodes are completely separate from streaming and serving nodes. Enabling the feature or changing the SKU/Replicas settings does not restart or interrupt your running streaming jobs.

**What happens if I set Replicas to 1 and submit 5 backfill jobs at the same time?**

The first eligible job runs immediately. The remaining jobs queue and run sequentially (or in parallel up to the replica count) as capacity becomes available. Jobs are not dropped.

**How do I choose a safe SKU and replica count?**

Start with 1 replica at a medium SKU and monitor how long backfills take. Increase replicas to run more jobs in parallel. Increase the SKU size if individual jobs are slow due to CPU or memory constraints.

**Is backfiller compute billed even when idle?**

No. When no backfill jobs are running, the backfiller is at 0 nodes and you are not charged for backfiller compute. You are charged only while nodes are active.
