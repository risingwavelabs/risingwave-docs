The compute cache is a local disk-based caching layer that sits between memory and object storage, reducing latency and object storage costs for performance-sensitive workloads on RisingWave Cloud.

<Note>
Compute cache is available on the **[Pro plan](/cloud/pricing#pro-plan)**, including fully hosted and BYOC deployments. The Basic plan does not support compute cache configuration.
</Note>

## When to use compute cache

The compute cache is beneficial when your workload's active data (the "working set") exceeds the available memory on your compute nodes. Without a disk cache, every cache miss must be served from object storage, which adds latency and increases costs. Common scenarios where compute cache helps:

- **Large-state streaming jobs**: Joins, time windows, and aggregations that generate internal state larger than available memory.
- **Data serving with frequent reads**: Queries against tables or materialized views where the same data is accessed repeatedly.
- **Frequent scaling operations**: The cache reduces warm-up time when new nodes are added, since they can read from local disk instead of object storage.

Key benefits:

- **Reduced object storage costs**: Fewer object storage reads by serving data from local disk.
- **Lower query latency**: Data served from local disk instead of remote object storage.
- **Faster scaling and recovery**: New or restarted nodes warm up quickly from local disk rather than refetching from object storage.

## View current configuration

To view the current compute cache configuration:

1. Navigate to the project overview page.
2. Click the `...` menu on the right, and select **Rescale**.

The current compute cache settings are displayed in the rescale dialog, showing:

- **Cache size (GB)**: The total disk capacity allocated for caching.
- **Performance tier**: The I/O performance level of the cache.

## Configure compute cache

### Cache size

You can adjust the cache size in gigabytes. The minimum size is **20 GB** and the step size is **1 GB**. The maximum size available for your project is shown in the rescale dialog.

Choose a cache size based on the working set of your streaming jobs. A larger cache reduces the frequency of object storage access, but comes with higher storage costs. As a general guideline, the default cache size scales with your compute node configuration: approximately 32 GB per [RWU](/cloud/pricing#risingwave-unit-rwu) for larger nodes (5 RWUs or more), or 100 GB total for smaller nodes.

### Performance tiers

RisingWave Cloud offers up to three performance tiers that control the I/O throughput and IOPS of the cache. The available tiers depend on your cloud provider and region:

| Tier            | Description                                                                 |
|:----------------|:----------------------------------------------------------------------------|
| **Standard**    | Baseline throughput and IOPS, suitable for most streaming and serving workloads. |
| **Performance** | Higher throughput and IOPS for latency-sensitive workloads with high read/write rates. |
| **Ultra**       | Maximum throughput and IOPS for the most I/O-intensive workloads. |

<Note>
Not all tiers are available in every region or cloud provider. The rescale dialog shows only the tiers available for your project. For example, some regions may only offer the Standard tier.
</Note>

The default tier for new projects is **Standard**. Upgrading the performance tier increases the hourly cost.

## Scale the compute cache

To update the compute cache configuration:

1. Navigate to the project overview page.
2. Click the `...` menu on the right, and select **Rescale**.
3. In the rescale dialog, adjust the **Compute cache size** and **Performance tier** as needed.
4. Click **Confirm** to apply the changes.

The update is applied asynchronously. Your project will briefly enter an updating state while the new configuration takes effect. Compute nodes may be restarted during this process, which can cause brief interruptions to in-progress queries. Streaming jobs recover automatically after the restart.

<Note>
You can update the compute cache as part of a compute node rescale operation. When rescaling compute nodes, the cache configuration can be adjusted in the same request.
</Note>

## Cache size recommendations

RisingWave Cloud provides a recommendation API that suggests an appropriate compute cache size based on your current compute node configuration. The suggested size is tuned to balance performance and cost.

<Note>
Cache size recommendations are available via the API. UI support for displaying recommendations in the rescale dialog is planned for a future release.
</Note>
